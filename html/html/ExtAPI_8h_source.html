47 
48 ** Specifications in ExtAPI.json
49 
50 *** [1] Overview of the Specification Language
51 The specification language of external functions is based on the JSON format. And every function defined by the Specification Language is an object that represents the specification rules. These Specification Language objects for functions contain four parts:
52 1. the name of the function,
53 2. the switch that controls whether the specification rules defined in the ExtAPI.json overwrite the functions defined in the user code,
54 3. the type of the function,
55 4. the operations conducted by the function.
56 
57 *** [2] Overwriting the user-defined functions
58 The switch *overwrite_app_function* controls whether the specification rules defined in the ExtAPI.json overwrite the functions defined in the user code (e.g., CPP files). When the switch *overwrite_app_function* is set to a value of 1, SVF will use the specification rules in ExtAPI.json to conduct the analysis and ignore the user-defined functions in the input CPP/bc files.
59 overwrite_app_function = 0: Analyze the user-defined functions.
60 overwrite_app_function = 1: Use specifications in ExtAPI.json to overwrite the user-defined functions.
61 
62 *** [3] Function types
63 Function type represents the properties of the function.
64 For example,
65 "EFT_ALLOC" represents if this external function allocates a new object and assigns it to one of its arguments,
66 For the selection of function type and a more detailed explanation, please refer to the definition of enum *extf_t* in ExtAPI.h.
67 
68 *** [4] Function operations
69 Function operations indicate the relationships between input and output,
70 mainly between function parameters or between parameters and return values after the execution.
71 For example,
72 "copy": ["A2", "L"] indicates that after this external function is executed, the value of the 2nd parameter is copied into the return value.
73 For the selection of function type and a more detailed explanation, please refer to the definition of enum *extType* in ExtAPI.h.
74 
75 For operands of function operation, e.g., "A2", "L", there are the following options:
76 "A": represents a parameter;
77 "N": represents a number;
78 "R": represents a reference;
79 "L": represents a return value;
80 "V": represents a dummy node;
81 
82 Among them, a parameter may have multiple forms because there may be various parameters, and the parameter may be a reference or complex structure,
83 
84 Here we use regular expressions "(AN)(R|RN)^*" to represent parameters, for example,
85 "A0": represents the 1st parameter;
86 "A2R": represents that the 3rd parameter is a reference;
87 "A1R2": represents the 3rd substructure of the 2nd parameter "A1R", where "A1R" is a complex structure;
88 "A2R3R": represents the 4th substructure of the 3rd parameter "A2R" is a reference, where "A2R" is a complex structure;
89 
90 
91 ** Specification format:
92 "functionName": {
93 "type": "functional type",
94 "overwrite_app_function:" 0/1,
95 "function operation_1": [ operand_1, operand_2, ... , operand_n],
96 "function operation_2": [ operand_1, operand_2, ... , operand_n],
97 ...
98 "function operation_n": [ operand_1, operand_2, ... , operand_n]
99 }
100 */

119  // External Function types
120  // Assume a call in the form LHS= F(arg0, arg1, arg2, arg3).
121  enum extType
122  {
123  EFT_NOOP = 0, // no effect on pointers
124  EFT_ALLOC, // returns a ptr to a newly allocated object
125  EFT_REALLOC, // like L_A0 if arg0 is a non-null ptr, else ALLOC
126  EFT_FREE, // free memory arg0 and all pointers passing into free function
127  EFT_FREE_MULTILEVEL, // any argument with 2-level pointer passing too a free wrapper function e.g., XFree(void**) which frees memory for void* and void**
128  EFT_NOSTRUCT_ALLOC, // like ALLOC but only allocates non-struct data
129  EFT_STAT, // retval points to an unknown static var X
130  EFT_STAT2, // ret -> X -> Y (X, Y - external static vars)
131  EFT_L_A0, // copies arg0, arg1, or arg2 into LHS
132  EFT_L_A1,
133  EFT_L_A2,
134  EFT_L_A8,
135  EFT_L_A0__A0R_A1, // stores arg1 into *arg0 and returns arg0 (currently only for memset)
136  EFT_L_A0__A0R_A1R, // copies the data that arg1 points to into the location
137  EFT_L_A1__FunPtr, // obtain the address of a symbol based on the arg1 (char*) and parse a function to LHS (e.g., void *dlsym(void *handle, char *funname);)
138  // arg0 points to; note that several fields may be
139  // copied at once if both point to structs.
140  // Returns arg0.
141  EFT_A1R_A0R, // copies *arg0 into *arg1, with non-ptr return
142  EFT_A3R_A1R_NS, // copies *arg1 into *arg3 (non-struct copy only)
143  EFT_A1R_A0, // stores arg0 into *arg1
144  EFT_A2R_A1, // stores arg1 into *arg2
145  EFT_A4R_A1, // stores arg1 into *arg4
146  EFT_L_A0__A2R_A0, // stores arg0 into *arg2 and returns it
147  EFT_L_A0__A1_A0, // store arg1 into arg0's base and returns arg0
148  EFT_A0R_NEW, // stores a pointer to an allocated object in *arg0
149  EFT_A1R_NEW, // as above, into *arg1, etc.
150  EFT_A2R_NEW,
151  EFT_A4R_NEW,
152  EFT_A11R_NEW,
153  EFT_STD_RB_TREE_INSERT_AND_REBALANCE, // Some complex effects
154  EFT_STD_RB_TREE_INCREMENT, // Some complex effects
155  EFT_STD_LIST_HOOK, // Some complex effects
156 
157  CPP_EFT_A0R_A1, // stores arg1 into *arg0
158  CPP_EFT_A0R_A1R, // copies *arg1 into *arg0
159  CPP_EFT_A1R, // load arg1
160  EFT_CXA_BEGIN_CATCH, //__cxa_begin_catch
161  CPP_EFT_DYNAMIC_CAST, // dynamic_cast
162  EFT_NULL // not found in the list
